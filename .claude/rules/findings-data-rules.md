# Findings & Dashboard — Стоп-правила

Данные отделены от вида: `index.html` рендерит фиксированный набор полей `findings.json`.
Любая поломка схемы или разметки = пустой/сломанный дашборд на проде.

## НИКОГДА

- **НЕ менять структуру `index.html`** и набор полей, которые он читает (см. контракт ниже). Правки
  вёрстки/стилей — только если явно просят и без изменения контракта данных.
- **НЕ менять схему `findings.json`**: top-level ключи (`schema`, `updated`, `excluded_niches`, `count`,
  `findings`) и форму элемента `findings[]`. Поле `schema` = `idea-radar-findings-v2` не трогать.
- **НЕ считать Diamond Score руками** и НЕ записывать его в данные — его вычисляет `index.html` из осей `s{}`.

## Контракт элемента `findings[]`

Обязательные поля каждого элемента (их читает дашборд):
`id`, `niche`, `method`, `conf`, `kano`, `status`, `title`, `segment`, `jtbd`, `pain`, `existing`, `idea`,
`s{severity, frequency, wtp, reach, gap, trend, ease}` (каждая ось — целое 1–5), `ev[{q, u, t}]` (≥1 элемент).
`demand` — опционально (для метода «спрос»).

Поля прослеживаемости прогонов (обязательны для НОВЫХ находок — дашборд использует их для фильтра прогонов и
отметок «новое/просмотрено»):
- `run` — id прогона, в котором находка добавлена. Формат `YYYY-MM-DD`; при нескольких прогонах в один день —
  суффикс `a`/`b`/`c` (`2026-06-27a`, `2026-06-27b`).
- `added` — дата добавления (`YYYY-MM-DD`).

Top-level ключ `runs` — реестр прогонов: массив `{id, date, label, count}` (по одному на прогон, в порядке
добавления). Дашборд строит из него выпадающий фильтр «Прогон». Поле `schema` остаётся `idea-radar-findings-v2`
(`run`/`added`/`runs` — аддитивное расширение; существующие чтения дашборда не ломаются).

## Файлы-спутники (внешняя память — экономия токенов)

`findings.json` большой (~120K токенов). **НЕ загружай его целиком в контекст** и **НЕ правь руками** — это
дорого и провоцирует повторы. Все мутации и чтение — через CLI `.claude/scripts/radar.py`, который держит в
синхроне два производных/сопутствующих файла:

- **`findings-index.json`** (`idea-radar-index-v1`) — производный компактный индекс: на каждую находку
  `{id, n(niche), m(method), st(status), sc(Diamond Score), run, k(концепт-ключ), t(title)}`. Это «внешняя
  память» прогона: дедуп и планирование идут по индексу, а не по большому файлу. Регенерируется `radar.py index`
  (и автоматически после `add`/`dismiss`/`merge`/`set-*`). В данные дашборда НЕ входит. `sc` — производная осей
  (та же формула `diamond()`), в `findings.json` его писать по-прежнему НЕЛЬЗЯ.
- **`dismissed.json`** (`idea-radar-dismissed-v1`) — блоклист отвергнутых идей: `{orig_id, niche, method, title,
  k, why, from_run, dismissed}`. Сюда уходят 👎«мимо» (и явно отвергнутые темы); из `findings.json` они
  УДАЛЯЮТСЯ. Прогон сверяется с блоклистом (`radar.py check` → `⛔ DISMISSED`) и НЕ предлагает их снова. Это
  блоклист, а не архив — полные данные удалённых находок остаются в истории git.

`max_id` в индексе — high-water id: новые находки получают `max_id+1`, **id удалённых/dismissed не
переиспользуются** (стабильность ссылок отметок вкуса).

Перечисления (ровно эти значения):

- `method`: `спрос` | `боль` | `идея`
- `conf`: `high` | `medium` | `low`
- `kano`: `basic` | `performance` | `delight`
- `status`: `NEW` | `REPEAT` | `GROWING` | `IDEA`
- `ev[].t` — **свободная** короткая метка типа источника (не закрытый список; дашборд рендерит её как текст ссылки).
  Типичные в данных: `стор`, `маркетплейс`, `форум`, `обсуждение`, `отзыв`, `github`, `статья`, `новости`, `обзор`, `отчёт`.

## При добавлении находки (через `radar.py`, НЕ руками)

1. **Дедуп ПЕРЕД добавлением:** `radar.py check "<3–6 слов сути>" --niche "<ниша>"`. Дедуп идёт по
   **фразонезависимому концепт-ключу** (нормализованные токены `title+segment+jtbd+idea+pain`, overlap-коэффициент,
   без эмбеддингов), а НЕ по «нормализованный `title` + `niche`» — старый ключ пропускал переформулированные
   дубли (одна и та же идея с другими словами и слегка иной нишей) и засорял дашборд повторами.
   - `⛔ DISMISSED` → DROP (тема отвергнута).
   - `⚠ ВЕРОЯТНЫЙ ДУБЛЬ` → не плодить новую; `radar.py set-status --ids <id> --status GROWING|REPEAT`.
   - иначе → NEW.
2. **Добавление:** запиши новые находки (без `id`/`run`/`added`) в `new-findings.json`, затем
   `radar.py add new-findings.json --label "<тема>"`. Скрипт проставит `id` (=`max_id+1`), `run`, `added`,
   обновит `count`/`updated`/`runs` и пересоберёт индекс.
3. **Проверка перед коммитом:** `radar.py validate` → `OK` (схема + согласованность индекса). Невалидный JSON
   ломает дашборд молча.

## Зачем

`index.html` обращается к полям напрямую (`d.s.gap`, `d.ev`, `d.niche`, `d.method`, …). Отсутствующее или
переименованное поле = `undefined` в UI или пустая карточка. Diamond Score — производная осей `s{}`;
дублировать его в данных нельзя — он рассинхронизируется с весами дашборда.
